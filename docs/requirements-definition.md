## 2. 目的

スーパーチャージャー到着時に空きストールがない利用者が、施設周辺で自由に過ごしながら公平に順番を待てる仕組みを提供する。

本MVPではTeslaアカウント連携、利用者アカウント、ログイン、電話番号収集、車両データの自動取得を行わず、ニックネームだけで参加できる利用者参加型の待ち列を実現する。通知は画面内通知を基本とし、希望者には匿名のWeb Pushを提供する。

## 3. 背景と解決する課題

### 3.1 現状の課題

- 現地が満車の場合、到着順を客観的に確認できない。
- 車内や周辺施設で待つと、自分の順番を把握しづらい。
- 順番を待つ車両同士で直接調整する必要があり、トラブルになりやすい。
- 充電終了予定が共有されないため、待ち時間を予測しにくい。

### 3.2 提供価値

- ニックネームだけで短時間に待ち列へ参加できる。
- 自分の前にいる人数と推定待ち時間を確認できる。
- 順番の5分前と順番到来を画面上で確認できる。
- 充電予定時間と完了報告を次の利用者の待ち時間へ反映できる。

## 4. サービスの位置づけ

- 本サービスはTesla公式またはTesla公認のサービスではない。
- 待ち列、空き状況、待ち時間は本サービス利用者の申告をもとにした参考情報である。
- 現地の交通ルール、施設ルール、Tesla車内表示を優先する。
- 本サービス上の順番は、物理的なストール利用権を法的・契約的に保証するものではない。

## 6. 想定利用者

### 6.1 待機利用者

- スーパーチャージャーへ到着したが空きがない。
- Google検索などから本サービスへアクセスする。
- アカウント作成なしで待ち列へ参加したい。
- 車内または周辺施設で待ちながら順番を確認したい。

### 6.2 充電利用者

- 待ち列から呼ばれて充電を開始する。
- おおよその充電予定時間を入力する。
- 充電終了時に完了を報告する。

## 7. 基本ユーザーフロー

画面一覧、画面ごとの表示・操作、状態遷移は`docs/screen-flow-spec.md`を正とする。

1. 利用者が現地で満車を確認する。
2. Google検索などから本サービスへアクセスする。
3. 施設名・地域名・住所の自由入力検索から対象施設を選択する。
4. 待ち人数と推定待ち時間を確認する。待ち列がなければ「待ち時間0分」「現地に空きがあればそのまま充電してください」と表示する。
5. ニックネームを入力して待ち列へ参加し、必要であれば「通知を受け取る」からWeb Pushを有効にする。この操作で利用者アカウントやログイン状態は作成しない。
6. 車内または周辺施設で待機する。
7. 画面上で「前にいる人数」と「推定待ち時間」を確認する。
8. 順番の約5分前に画面内で通知され、Web Pushを有効にした利用者には端末通知も送信される。
9. 順番になると5分の受付時間が開始する。
10. 利用者は5分以内に「充電を開始しました！」を押す。
11. 5分以内に操作しない場合は待ち列から自動削除される。
12. 充電開始後、予定充電時間を入力する。
13. 充電終了後、「充電が終わりました」を押す。
14. 次の利用者の順番と推定待ち時間が更新され、完了した利用者の待ち列データは削除される。

## 9. 状態定義

| 状態 | 説明 | 主な遷移先 |
|---|---|---|
| `waiting` | 待ち列で待機中 | `notified`, `cancelled` |
| `notified` | 順番の約5分前 | `called`, `cancelled` |
| `called` | 順番到来、5分受付中 | `charging`, `expired`, `cancelled` |
| `charging` | 充電中 | `completed`, `cancelled` |
| `completed` | 充電完了処理中の一時状態 | 後続を更新して行削除 |
| `cancelled` | 利用者取消処理中の一時状態 | 後続を更新して行削除 |
| `expired` | 呼び出し後5分以内に応答なし | 後続を更新して行削除 |

## 10. 業務ルール

| ID | ルール |
|---|---|
| BR-001 | 原則として、施設ごとの参加時刻が早い順（FIFO）で処理する。 |
| BR-002 | 利用者が取消・失効した場合、後続順位を即時繰り上げる。 |
| BR-003 | 呼び出し枠の有効時間は5分とする。 |
| BR-004 | 同一ブラウザの有効エントリーはローカル保存した管理トークンで復元する。アカウントを作らないため、端末をまたいだ同一人物の判定は行わない。 |
| BR-005 | ニックネームの重複は許容し、画面内部では公開IDで識別する。 |
| BR-006 | 待ち時間は確約値ではなく、充電中利用者の予定終了時刻、ストール数、待ち人数から算出した参考値とする。 |
| BR-007 | 完了報告がない場合は予定時刻超過を画面へ表示する。運用者補正を設けないため、自動解放の猶予時間は実装前に確定する。 |
| BR-008 | 充電開始・完了・取消・失効の処理後は、施設単位で順位と待ち時間を再計算する。 |
| BR-009 | 待ち列参加者の正確な位置、移動経路、車両識別情報は他の利用者へ表示しない。 |
| BR-010 | 画面を閉じた場合でも待ち列は維持する。画面表示の自動更新は接続状況に依存するが、Web Pushを有効にした利用者には5分前・順番到来・充電終了確認の端末通知を送信する。 |
| BR-011 | Web Pushは通知を許可したブラウザSubscriptionにのみ送信し、同一待ち列へ同一種類の通知を重複送信しない。 |
| BR-012 | 複数ストールが同時に空いた場合、その数と同数までFIFO先頭から同時に`called`へ遷移させる。 |
| BR-013 | 完了・取消・失効したエントリーは履歴テーブルへ移さず、後続計算後に即時削除する。再利用時はニックネームを再入力し、必要であればWeb Pushを再度有効化する。 |
| BR-014 | 管理トークン原文はDBへ保存せずハッシュだけを有効エントリー行に保存し、終了時に行ごと削除する。 |
| BR-015 | `waiting/notified/called`が存在しない施設の待ち時間は0分とし、現地の各充電車へ終了予定の登録を要求しない。 |
| BR-016 | アプリ上の有効な待機者が0人の状態で、利用者が現地満車を確認して参加した時点を待ち列開始時刻とする。アプリ利用中の充電車に確定終了時刻があればそれを使い、それ以外のストールは暫定45分で初期化する。 |
| BR-017 | アプリの待ち列と現地状況が異なる場合は現地の安全・施設ルール・実際の並び順を最優先する。アプリ表示は利用者申告による参考値であり、利用権を保証しない。 |
| BR-018 | 予定終了時刻の3分前に延長・終了の確認を表示する。「まだ充電する」が選ばれた場合は追加時間で再計算する。予定終了時刻の5分後まで無応答なら自動完了し、後続を繰り上げる。 |

## 11. 待ち時間の考え方

### 11.1 基本方針

- 充電中利用者が入力した予定終了時刻を優先する。
- 予定終了時刻が不明なストール、およびまだ充電時間を入力できない待機者には、標準充電時間45分を使用する。
- 施設のストールごとに「次に空く予定時刻」を持つ。
- 待ち列の各利用者を、次に空く予定が最も早いストールへFIFO順に仮割り当てする。
- 仮割り当て後の次回空き予定には、その待機者の予定充電時間が未入力のため、施設別の標準充電時間を加算する。
- 完了、取消、初回の予定時間確定、終了予定3分前の延長確定のたびに再計算する。
- 画面には1分単位の厳密値ではなく、5分単位へ丸めた目安を表示する。
- 待機者が実際に充電開始し自由入力した予定時間は、その時点から標準充電時間より優先する。

### 11.2 基本計算例（3ストール）

現在時刻が15:20、次回空き予定が次の3枠で、待機者の標準充電時間を45分とする。

```text
ストール1：15:30に空く
ストール2：15:45に空く
ストール3：15:45に空く
```

| 待機者 | 仮割り当て | 表示 | 割り当て後の次回空き |
|---|---|---|---|
| A | ストール1・15:30 | あと10分 | 16:15 |
| B | ストール2・15:45 | あと25分 | 16:30 |
| C | ストール3・15:45 | あと25分 | 16:30 |
| D | 最も早く再度空くストール1・16:15 | あと55分 | 17:00 |

したがって、ご提示の例が3ストールならDは暫定で「あと55分」となる。45分はAの予定時間がまだ不明な場合だけ使う仮定値であり、Aが開始時に20分と入力すればDは15:50開始の「あと30分」へ更新される。Aが60分と入力した場合はストール1の次回空きが16:30となり、Dはストール1〜3のいずれかへ16:30開始の「あと70分」として再計算される。

### 11.3 4ストールの場合

4ストール施設では、Dを計算するために4つ目のストールの次回空き予定が必要になる。

- 4つ目が15:50に空く場合：Dはあと30分
- 4つ目が15:45に空く場合：B、C、Dはいずれもあと25分
- 4つ目が現在空いている場合：Aは即時、Bはあと10分、CとDはあと25分
- 4台すべてが15:45に終了した場合：A〜Dを15:45に同時呼び出しする

4つ目の終了予定が未入力の場合、Dは一意に決められない。画面上では施設別の標準値を使った「暫定予測」として表示するか、「計算中」と表示する。

### 11.4 複数台が同時に空いた場合

- 同一時刻に4ストールが空けば、待ち列の先頭4人を1つのトランザクションで同時に呼び出す。
- 4人それぞれに独立した5分の受付期限を設定する。
- 1人だけ失効した場合は、その1枠について次の待機者を繰り上げる。
- 完了報告がほぼ同時に複数届いた場合も、空いた枠数を合算して一度に繰り上げる。

### 11.5 表示例

- 「約25分」
- 「前に3人」
- 「4台が同時に空く見込み」
- データが不足する場合は「暫定：約45分」「計算中」「目安を算出できません」を使い分ける。

## 12. 通知要件

### 12.1 MVP

- 待機画面を開いている利用者について、順位、前方人数、待ち時間、状態をリアルタイムで自動更新する。この自動更新自体は利用者向け通知ではなく、音・振動・Web Pushを発生させない。
- 順番の約5分前と順番到来時に、色・文言・音・可能な端末では振動で明確に通知する。
- Web Pushは任意とし、待ち列参加後の利用者操作を起点にブラウザの通知許可を要求する。初回アクセス直後には要求しない。
- OneSignalが発行するブラウザ・端末単位の匿名Subscription IDを有効な待ち列行へ一時的に紐付ける。利用者アカウントは作成しない。
- Web Pushを有効にした利用者へ、推定呼び出し時刻の約5分前、順番到来、予定充電終了時刻の3分前の延長・終了確認を通知する。
- 順番到来通知は、通常の空き発生と、前の利用者が呼び出し後5分以内に充電開始せず繰り上がった場合を同じ通知種類として扱う。
- 待ち列参加、取消、順位変更、初回の充電時間確定・延長確定だけを理由とするWeb Pushは送らず、画面表示だけを更新する。
- 再計算によって条件を複数回またいでも、同一待ち列に同一種類のPushを重複送信しない。
- Push本文には施設名、行動案内、待機画面URLを含め、ニックネームや他利用者情報は含めない。
- iPhone/iPadではiOS/iPadOS 16.4以降かつホーム画面へ追加したWebアプリである必要があるため、対象端末だけ手順を案内する。
- 充電終了予定時刻の3分前には充電利用者へ延長・終了の画面内確認を表示し、予定終了時刻の5分後に自動完了した場合も結果を表示する。
- 自動完了または利用者の完了報告でストールが空いた扱いになった場合、次の待機者を即時に呼び出して画面を更新する。

### 12.2 将来対応

- 通知の再送、通知カテゴリ、言語設定を拡張する。
- 到達性が必要になった場合のみSMSまたはネイティブアプリを再検討する。

## 13. 非機能要件

| 分類 | 要件 |
|---|---|
| 対応端末 | iOS Safari、Android Chromeを主対象とする。PCでも主要操作が可能であること。 |
| 応答性能 | 通常APIは95パーセンタイルで1秒以内、画面遷移は体感2秒以内を目標とする。 |
| リアルタイム性 | 待ち列変更を通常5秒以内に画面へ反映する。 |
| 可用性 | MVPは月間99.5%以上を目標とする。 |
| アクセシビリティ | 色だけに依存せず、キーボード操作とスクリーンリーダー用ラベルを提供する。 |
| セキュリティ | 通信はVercelとSupabaseが提供するHTTPSを使用する。管理トークン原文とOneSignal REST API KeyをDB・ブラウザ公開領域へ保存しない。 |
| プライバシー | 電話番号、メール、Supabase Auth利用者、VIN、正確な位置履歴を収集しない。Push Subscription IDは有効な待ち列との一時的な紐付けだけに使用する。 |
| データ保持 | 完了・取消・失効時に個別エントリーとPush Subscription IDの紐付けを即時削除し、履歴・利用者プロファイル・通知履歴を別テーブルへ残さない。OneSignalやバックアップ上の保持は各事業者のポリシーに従う。 |
| 監査 | MVPでは利用者単位の永続監査ログを作成しない。サービス障害調査用ログへニックネーム、Push Subscription ID、管理トークンを出力しない。 |
| バックアップ | データベースを日次バックアップし、7日以上保持する。 |

## 14. 受入条件

### AC-001 待ち列参加

- Given: 施設の待ち列が有効である
- When: 利用者が有効なニックネームを入力して参加する
- Then: 利用者がFIFOの末尾へ追加され、順位と推定待ち時間が表示される

### AC-002 重複参加防止

- Given: 同一端末が同一施設の有効な待ち列へ参加済みである
- When: 再度参加しようとする
- Then: 新規エントリーを作らず、既存の待機画面へ案内する

### AC-003 5分前表示

- Given: 利用者の推定呼び出し時刻まで5分以内である
- When: 待ち時間が再計算される
- Then: 状態が`notified`となり、施設付近へ戻る案内が表示される

### AC-004 呼び出し失効

- Given: 利用者が`called`状態である
- When: 5分以内に充電開始を報告しない
- Then: 状態が`expired`となり、後続の利用者が繰り上がる

### AC-005 充電完了

- Given: 利用者が`charging`状態である
- When: 「充電が終わりました」を押す
- Then: 待ち時間が再計算され、完了したエントリーがDBから削除される

### AC-006 自主退出

- Given: 利用者が`waiting`または`notified`状態である
- When: 確認後に待ち列から退出する
- Then: 待ち時間が再計算され、取消したエントリーがDBから削除される

### AC-007 充電時間の自由入力

- Given: 利用者が充電を開始している
- When: 5〜120分の整数を入力して確定する
- Then: 終了予定時刻と後続利用者の推定待ち時間が再計算される

### AC-008 Web Push

- Given: 利用者がブラウザ通知を許可し、Web Pushを有効にしている
- When: 推定呼び出し時刻まで5分以内になる
- Then: 画面内通知に加えてWeb Pushが1回だけ送信され、送信済み時刻が有効エントリーへ記録される

### AC-009 待ちなし

- Given: 施設に`waiting/notified/called`の利用者が存在しない
- When: 施設詳細を表示する
- Then: 待ち人数0人・待ち時間0分と、「現地に空きがあればそのまま充電してください」を表示する

### AC-010 待ち列の新規開始

- Given: 待機者が存在せず、利用者が現地満車を確認している
- When: 利用者が待ち時間0分の施設で、新たに待ち列を開始する
- Then: 確定終了時刻がないストールを現在時刻から45分後に空く暫定値として待ち時間を計算する

### AC-011 予定終了前の延長確認

- Given: 充電利用者の確定した予定終了時刻まで3分になった
- When: 延長・終了確認が表示された後、予定終了時刻の5分後までに追加時間も完了も選ばない
- Then: エントリーを自動完了し、次の利用者を呼び出す。待機者がいる場合だけ移動を促す強い文言を表示する

## 15. 成功指標

- 待ち列参加完了率
- 参加から充電開始までの中央値
- 呼び出し後5分以内の応答率
- 自主退出率、失効率
- 充電完了報告率
- Web Pushの許可率、送信成功率、クリック率、重複送信率
- 推定待ち時間と実待ち時間の平均誤差
- 待ち列に関する問い合わせ・トラブル件数

## 16. 制約とリスク

| リスク | 内容 | MVPでの対応 |
|---|---|---|
| 自己申告漏れ | 充電完了が報告されない | 予定時刻超過の確認表示。運用画面を設けないため、自動解放の猶予時間を実装前に確定する |
| なりすまし・荒らし | ログインなしでは本人性と端末横断の重複防止が弱い | 推測困難な管理トークン、CAPTCHA、IP・施設単位レート制限、同一ブラウザの重複抑止 |
| 現地順との不一致 | アプリ未利用者との順番を調整できない | 参加型データであることを明示し、現地ルール優先とする |
| 通知未達 | 通知拒否、OS設定、集中モード、iPhoneでホーム画面未追加、通信障害 | 画面内通知の併用、対応条件の案内。Pushは補助であり順番確認は待機画面を正とする |
| Push Subscription無効化 | ブラウザデータ削除、権限取消、期限切れ | Push失敗でも待ち列を維持し、無効Subscriptionの紐付けを削除する |
| 待ち時間誤差 | 予定時間や空き状況が不正確 | 参考値表示、5分単位丸め、継続的な再計算 |
| 安全性 | 運転中の画面操作 | 停車中に操作する旨を明記し、操作を最小化 |

## 17. MVP開始前に確定すべき事項

1. 最初に対象とする施設数と地域
2. 施設情報の初期登録・更新責任者
3. 「ストールが空いた」と判定する情報源
4. 呼び出し対象数（空き1基につき1人でよいか）
5. 同一端末が同時参加できる施設数
6. 全施設共通の標準45分を、将来施設別に上書き可能とするか
7. 予定終了後も完了報告がない場合の自動解放猶予時間
8. [利用規約ドラフト](terms-of-service.md)の運営者情報・連絡先・適用日を確定し、プライバシーポリシーと公開前の専門家レビューを完了すること
9. OneSignal App、通知アイコン、通知許可前の説明文、外部事業者側のデータ保持設定
10. 施設マスタ変更用migrationの作成・レビュー・デプロイ手順
