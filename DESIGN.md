# スパQ — DESIGN.md

> この文書は、スパQ（Teslaスーパーチャージャーの待ち列アプリ）の見た目と操作感を定義する唯一のデザイン基準である。画面やコンポーネントを追加・変更するAIエージェントと開発者は、実装前に本書と`docs/screen-flow-spec.md`を読むこと。

## 1. Visual Theme & Atmosphere

### プロダクトの印象

- **落ち着いた、信頼できる、移動中でも迷わない待ち列体験**。
- 清潔な白とやわらかなセージ系の面を基調に、深いフォレストグリーンを主役にする。
- 混雑や時間切れの緊張は、必要な状態だけで明確に伝える。常時派手・常時警告にしない。
- データは読み取りやすく、行動に必要な数字を大きく、補助情報を小さく整理する。
- 現在のMockにある「丸い進捗・やわらかいカード・細い罫線・濃緑の行動ボタン」を継承する。

### 守るべき境界

- これはTesla公式アプリではない。Teslaのロゴ、独自書体、公式UIの模倣はしない。
- 実アプリではMockの端末フレーム、`MOCK`バッジ、左側の説明文、右側のフロー図を表示しない。アプリ内容だけを表示する。
- 位置情報、距離、地図、現在地ボタン、「近い順」は要件外であり表示しない。
- 「現地を最優先」「待ち時間は参考値」という注意書きは、待機・順番到来・充電中の画面から省略しない。

## 2. Color Palette & Roles

| Token | 値 | 用途 |
|---|---:|---|
| `--color-ink` | `#17201D` | 見出し、本文、主要な数字 |
| `--color-muted` | `#728078` | 補助文、ラベル、非強調の説明 |
| `--color-paper` | `#FFFFFF` | 基本サーフェス、カード |
| `--color-cream` | `#F1F1E9` | 背景のやわらかな面、控えめな区切り |
| `--color-page` | `#E7E9E3` | デスクトップの外側背景のみ |
| `--color-primary` | `#1B5948` | 主ボタン、ロゴ、主要数値、完了状態 |
| `--color-primary-hover` | `#104A3A` | 主ボタンのhover / pressed |
| `--color-energy` | `#CAEF86` | 充電・進捗・成功を補助するライム。面積は小さく使う |
| `--color-success` | `#178255` | 空きあり、正常状態のドット |
| `--color-warning` | `#D18718` | 5分前・注意が必要な状態 |
| `--color-warning-bg` | `#FFF4DC` | 5分前通知の面 |
| `--color-danger` | `#DD3B35` | 時間切れ、削除、破壊的操作。通常CTAには使わない |
| `--color-danger-bg` | `#FFF7F6` | 失効・期限警告の面 |
| `--color-line` | `#E2E6E1` | 境界線、区切り線 |
| `--color-subtle` | `#F4F7F4` | 説明カード、数値の背景 |

### 状態の色

- **待機中**: `primary`と中立色。緊急性を出さない。
- **5分前**: `warning` + `warning-bg`。黄色のバナーと短い行動案内。
- **順番到来**: `primary`の大きな円、`energy`のリング。主CTAは`danger`ではなく`primary`。
- **充電中・完了**: `primary`と`energy`。
- **失効・退出・削除**: `danger`は結果・補助操作に限定し、本文で理由を明示する。

## 3. Typography Rules

### Font stack

```css
font-family: Geist, "Hiragino Kaku Gothic ProN", "Yu Gothic", sans-serif;
font-family-mono: "Geist Mono", ui-monospace, monospace;
```

- 数字のカウントダウンや技術的な短い識別子だけに`font-family-mono`を使う。
- 日本語本文は読みやすさ優先。英字を不必要に全大文字へ変換しない。

### Type scale

| Role | Size / line-height | Weight | Letter spacing | 用途 |
|---|---|---:|---:|---|
| Page title | `25px / 1.15` | 700–800 | `-0.045em` | 「待ち列に参加」「充電しています」 |
| Section title | `15–16px / 1.3` | 700 | `-0.02em` | カード見出し、施設名 |
| Hero number | `24–54px / 0.95` | 700–800 | `-0.05em` | 順番、待ち時間、カウントダウン |
| Body | `11px / 1.65` | 400–500 | normal | 説明、注意書き |
| Label | `8–9px / 1.4` | 700–800 | normal | 数値ラベル、入力ラベル |
| Overline | `10px / 1.2` | 800 | `0.18em` | 状態の小見出し。英字のみ使用可 |
| Microcopy | `7–8px / 1.45` | 500–700 | normal | 補足、免責、カード内の短文 |

## 4. Component Stylings

### App shell & header

- モバイルでは白いアプリ面を全面に表示する。デスクトップでは最大幅`480px`のアプリ面を中央に置く。
- ヘッダーは高さ`48px`、白背景、下線`1px solid var(--color-line)`。
- 中央に小さいブランドマークと`スパQ`。戻る操作は左に`32px × 32px`の領域を確保する。
- 画面見出し・主要操作をヘッダーと競合させない。ヘッダーを情報過多にしない。

### Buttons

| 種類 | 見た目 | 使用場面 |
|---|---|---|
| Primary | `primary`背景、白文字、角丸`11px`、高さ約`48px` | 参加、開始、時間確定、延長確定、完了 |
| Secondary | 白背景、`primary`の1px枠、緑文字 | 戻る、補助的な確定 |
| Text / destructive | 背景なし、下線または控えめな文字 | 待ち列から退出、キャンセル |
| Disabled | 不透明度`0.35`、hoverなし | ニックネーム未入力、送信中 |

- タップ領域は最低`44px`。主ボタンは画面内で常に1つだけを最も強く見せる。
- 送信中はラベルを「参加しています…」などへ変え、二重タップを防ぐ。
- 破壊的操作を赤い塗りつぶし主ボタンにしない。確認モーダル内の二次操作として扱う。

### Inputs & chips

- テキスト入力は高さ`49px`、白またはごく薄い背景、1px枠、角丸`11px`。
- focusは`primary`の枠と、4pxの淡い緑の外周で示す。色だけでなくラベルとエラー文も必ず表示する。
- 充電時間のプリセットは4列までの同サイズチップ。選択中は淡い緑背景 + `2px solid primary`。
- 自由入力欄はプリセットと同じ画面に置き、「5〜120分」を常に明示する。

### Cards & panels

- 通常カード: 白、1pxの`line`、角丸`13–14px`、影は`0 3px 12px rgba(39,56,47,.04)`程度。
- 強調サマリー: `primary`背景、白文字、角丸`16px`。待ち人数と待ち時間を2列で大きく表示する。
- 説明カード: `subtle`背景、枠なしまたは薄い枠、角丸`10–13px`。
- 注意カード: 警告は`warning-bg`、期限切れは`danger-bg`。大きな赤面・強い影を乱用しない。

### Queue indicators

- 待機中は「現在○番目」を円形の`position-orbit`で表示する。外周は薄緑、中央数字は`primary`。
- 5分前は同じ構造を維持し、上部の黄色バナーと数値色の`warning`で状態を変える。
- 順番到来は大きな`primary`の円と`energy`のリング。リングのアニメーションはこの状態だけに使う。
- 充電中は小さめの円、残り時間カード、細い進捗バーを組み合わせる。バッテリー残量・kWなど実データがなければ表示しない。

### Modals & feedback

- 終了予定3分前の「終了 / 延長」確認は画面中央またはボトムシートのモーダルで表示する。
- モーダル背景は白、角丸`16px`以上、タイトル・短い説明・主操作・戻る操作を持つ。
- モーダル表示中はフォーカスを内部へ閉じ込め、Esc・閉じる操作・スクリーンリーダーラベルを提供する。
- Toastは保存成功や軽微な更新にのみ使う。順番到来や失効は専用状態画面で明確に示す。

## 5. Layout Principles

### Spacing & geometry

```text
4 / 8 / 12 / 16 / 20 / 24 / 32px
```

- 画面の左右paddingは標準`20px`、縦paddingは`22px`を基準にする。
- カード間は`8–12px`、主要セクション間は`16–24px`。
- 一画面につき、最重要数値は1つに絞る。待機画面なら順位、順番到来なら残り受付時間、充電中なら終了予定。
- 最大コンテンツ幅は`480px`。広い画面でデータカードを不必要に横へ広げない。

### Content hierarchy

1. 現在の状態
2. 利用者が今すべきこと
3. 重要な数値
4. 参考情報・現地優先の注意
5. 退出などの二次操作

「前にいる人数」「待ち時間」「推定呼び出し時刻」は同じ視覚グループに置く。ニックネーム、他人の個人情報、技術的な識別子を主情報より強く見せない。

## 6. Depth, Motion & Elevation

- 基本は1px罫線で面を分ける。影はカードの浮きをわずかに示すだけにする。
- 画面遷移は`opacity`と横`8px`以内の移動を`0.28s ease`で行う。
- 進捗バーは最大`1s`の滑らかな補間。
- 順番到来リングのみ、`1.8s`程度の淡いpulseを許可する。
- `prefers-reduced-motion: reduce`時は、すべてのアニメーションを実質無効化する。
- 点滅、常時揺れ、無意味な数値カウント、強いパララックスは使用禁止。

## 7. Do's and Don'ts

### Do

- 深い緑を「次へ進む行動」と「信頼できる状態」に一貫して使う。
- 時間や順番を大きく、単位を小さく表示する。
- 状態が変わった理由を一文で説明する。
- 色に加えて、文言・アイコン・位置で待機 / 注意 / 順番到来を区別する。
- 現地優先の注意を、待機・順番到来・充電中で常に確認できるようにする。

### Don't

- 現在地、距離、地図、Google Mapsへの誘導をUIへ追加しない。
- 未取得の充電出力、バッテリー残量、ストール番号をダミー表示しない。
- 通知許可を初回表示や検索中に要求しない。
- 充電中画面に「任意の充電時間変更」を置かない。延長は終了予定3分前の確認だけで行う。
- 画面ごとに新しい配色・角丸・影・フォントを発明しない。
- 複数の塗りつぶし主ボタンを同時に置かない。

## 8. Responsive Behavior & Accessibility

- **レスポンシブ対応はMVPの必須条件**とする。主対象は`390px`幅のモバイルであり、`320px`、`360px`、`390px`、`430px`の各幅で、文字・ボタン・数値が欠けずに操作できること。
- `700px`未満ではアプリを全画面表示し、端末フレームを使わない。
- `700px`以上では中央にアプリ面を置く。必要なら補足説明を外側へ置くが、アプリ操作を分断しない。
- 横スクロールを発生させない。キーボード表示時、長い施設名・住所・エラー文言・通知許可案内でも、主CTA、入力、モーダルの操作が隠れないようにする。
- 縦向き・横向きのいずれでも、重要な操作が画面外に固定されたり、セーフエリアやブラウザUIと重なったりしないようにする。
- スクロール可能な画面では、固定CTAが本文やホームインジケータと重ならないよう下部safe areaを確保する。
- コントラストは本文・ボタン・状態表示でWCAG AA相当を目安にする。
- アイコンだけのボタンへ`aria-label`を設定する。エラーは入力欄と関連付け、色だけで伝えない。

## 9. Agent Prompt Guide

### 必ず守る指示

```text
Read DESIGN.md and docs/screen-flow-spec.md before changing UI.
スパQの配色とコンポーネントルールを使用する。
Keep the app mobile-first, light, calm, card-based, and data-legible.
Do not add location, distance, maps, Tesla branding, arbitrary charging metrics, or an editable duration control during charging.
Use one clear primary CTA per screen. Preserve the required current-state, next-action, key-number hierarchy.
```

### 画面実装時の確認

- この画面の主状態は何か。
- 利用者の次の行動は1つに絞れているか。
- 既存tokenだけで構成できるか。
- 状態色が情報の意味と一致しているか。
- 現地優先・参考値の説明が必要な画面で消えていないか。
- `docs/screen-flow-spec.md`の操作・API・遷移と矛盾していないか。

## 10. Visual QA Checklist

- [ ] モバイル390px幅で、主CTA・主要数値・注意書きが一画面で認識できる。
- [ ] `320px`、`360px`、`390px`、`430px`幅で横スクロール、文字切れ、タップ不能な主CTAがない。
- [ ] モバイルの縦向き・横向き、ソフトウェアキーボード表示時にも、入力・エラー・主CTAを操作できる。
- [ ] 5分前、順番到来、失効、完了が色以外でも区別できる。
- [ ] すべての主ボタン、入力、モーダルがキーボードとスクリーンリーダーで操作できる。
- [ ] loading、送信失敗、Realtime再接続中でもレイアウトが崩れない。
- [ ] 画面ごとに余白、カード角丸、罫線、ボタン高、文字階層が揃っている。
- [ ] 本番UIにMock専用の端末フレーム・デモ操作・距離表示が残っていない。
